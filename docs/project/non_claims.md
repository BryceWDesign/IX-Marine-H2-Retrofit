# Explicit Non-Claims

## Purpose

This file exists to prevent IX-Marine-H2-Retrofit from being misunderstood.

The repository is intentionally ambitious in scope, but it is not allowed to
hide behind vague language. This file defines what the repository **does not**
claim so that readers can separate:

- concept from certification
- architecture from guarantee
- engineering direction from finished product
- proof-of-concept from operational release

## Core non-claim

IX-Marine-H2-Retrofit does **not** claim to have created a vessel that powers
itself indefinitely from seawater, wave action, intake flow, or vessel drag
alone.

The repository may explore auxiliary harvesting for controls and housekeeping
functions, but it does not present those functions as a substitute for the
primary energy required to generate propulsion-scale hydrogen.

## Certification and approval non-claims

This repository is **not**:

- a class-approved vessel package
- a flag-approved retrofit package
- a shipyard-issued installation release
- a port-authority approval package
- an insurer-approved risk package
- a final fire-response doctrine
- a substitute for formal HAZID, HAZOP, FMEA, or class review performed on a
  specific vessel by qualified parties

Nothing in this repository should be read as proof that a real owner, yard,
class society, or authority has accepted the architecture for live service.

## Performance non-claims

This repository does **not** claim:

- universal positive economic payback
- universal emissions superiority in every electricity supply scenario
- universal suitability for all vessel classes
- deep-sea endurance viability
- a final installed mass or footprint for every hull
- final storage, range, or berth-time figures for all operators
- a final hazardous-zone dimension that automatically applies to real vessels
- a final stability approval result for a real conversion

All numerical sizing in the repository is concept-stage and reference-vessel
specific unless a file explicitly says otherwise.

## Technology non-claims

This repository does **not** claim that:

- raw seawater can be fed directly into the selected electrolyzer as the
  baseline production path without treatment penalties
- hydrogen production onboard is inherently cheap
- compressed hydrogen storage volume is trivial in marine use
- small harvesters or weak ambient sources can replace primary vessel energy
  architecture
- one retrofit architecture solves every marine decarbonization case
- fuel-cell integration is simple, low-risk, or approval-light by default

## Safety non-claims

This repository does **not** claim to eliminate:

- hydrogen leak risk
- ignited release risk
- ventilation failure risk
- maintenance error risk
- seawater fouling risk
- compressor fault risk
- bad-water damage risk
- human-factor risk

The repository is built to expose and manage these risks at concept stage, not
to pretend they disappear.

## Market and adoption non-claims

This repository does **not** claim that:

- every shipowner should choose hydrogen
- hydrogen is always smarter than battery, HVO, methanol, ammonia, or other
  pathways
- owners will adopt this architecture based on fuel savings alone
- ports are already ready everywhere
- training burden is small
- retrofit downtime is negligible

Where hydrogen appears attractive in this repository, it is tied to a selected
harbor / short-sea vessel use case and a strategic decarbonization framing.

## Operational non-claims

The repository does **not** claim that onboard electrolysis will normally be
the sole replenishment path.

The default framing remains:

- onboard hydrogen generation primarily while berthed
- dependence on external electrical power
- possible backup shore hydrogen loading
- battery support as a core part of vessel operability

The repository also does **not** claim that underway hydrogen generation from
ship-motion-derived energy is the primary operating mode.

## Assembly and build non-claims

Any bill of materials, assembly guidance, or integration walkthrough included
in this repository is concept-stage only.

Those materials are **not**:

- a fabrication release
- a stamped drawing package
- a live-service commissioning procedure
- a certified maintenance manual
- a guarantee of safe build sequence on a real vessel

They are included to make the architecture inspectable, not to bypass yard,
class, or marine safety process.

## Relationship to proof-of-concept work

Proof-of-concept artifacts in this repository are intended to answer questions
such as:

- can the architecture be reasoned through coherently
- are the interfaces understandable
- are the major hazards surfaced
- can a reviewer trace assumptions to design choices

Proof-of-concept artifacts are **not** presented as full-scale operational
validation.

## Reader instruction

If any later file sounds stronger than this file allows, this file wins unless
that later file provides a more bounded and evidence-backed statement.

This repository is strongest when its limits are visible.
